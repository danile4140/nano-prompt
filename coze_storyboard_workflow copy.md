# 1. 文档说明

这是一个用于将英文原著文章拆解为可生成插图/绘本分镜提示词的工作流文档。目标是帮助儿童能够基于图片完整、自然、连贯地复述整个故事。
本工作流当前主要包含 3 个主节点：

- 5.1 角色生成节点
- 5.2 场景生成节点
- 5.3 分镜生成节点（按章节循环）
此外还有两个辅助节点：
- 5.3.5 分镜校验节点：对 5.3 产出的分镜做对齐校验并**直接输出修正后的 storyboard_assets**，不输出问题清单，可作为 5.3 的下游修复层
- 5.4 分镜调整节点：根据人工调整意见对分镜做二次微调
本版文档针对 **Nano Banana 2** 做了专门优化。核心目标是**让孩子看图复述原文**，因此每张分镜都要尽可能高密度地承载原文信息。设计哲学是：

1. 先保证故事理解正确
2. 再保证角色和场景可稳定复用
3. 在角色/场景稳定的前提下，用多格构图、对白气泡、思维气泡、画中画、角标等漫画语言，把单张图变成"独立可复述的叙事节拍"
补充: nano banana 的官方提示词指南  <https://nanobanana.io/zh/prompt-guide>

# 2. Nano Banana 2 适配原则

## 2.1 总体原则

Nano Banana 2 更适合：

- 主体清晰
- 动作明确
- 场景明确
- 重点前置
- 短句化表达
- 稳定复用固定描述
Nano Banana 2 不适合：
- 单条 prompt 叠加过多任务目标
- 单图同时承载过多动作链和因果链
- 多层构图元素同时堆叠
- 角色描述前后频繁变形
- 只写氛围、不写空间结构和比例参照
因此，本工作流采用以下生成哲学：

> **故事完整性靠拆镜+多格构图共同保证；画面稳定性靠固定描述复用保证；复述价值靠单图高密度叙事保证。**
每张分镜应当承担一段"独立可复述的叙事节拍"，即：孩子看到这张图，不需要看上一张或下一张，就能复述出原文中对应的那一段内容（包括动作、情绪、关键道具、对白要点）。
---

## 2.2 提示词总结构

所有最终送入 Nano Banana 2 的分镜 prompt，优先遵循以下结构：

1. 主体与角色
2. 动作与关系
3. 场景（地点/时间/天气/布局）
4. 风格与媒介
5. 构图与镜头
6. 光影与色彩
7. 关键细节
8. 约束条件
重要信息必须放在前面。不要把系统规则、流程说明、统计要求写进最终生图 prompt。

---

## 2.3 分镜设计原则

1. **逐句对应**：原文中的每一句话，都必须能在分镜序列中找到对应画面。
2. **高密度叙事**：单张分镜要在角色/场景稳定的前提下，尽可能多承载原文信息。允许并鼓励使用多格构图、对白气泡、思维气泡、画中画、角标等手段，把"一段原文"压缩进"一张图"。
3. **节拍而非动作**：拆镜的最小单位是"叙事节拍"（一段语义完整、孩子能独立复述的剧情），而不是"单个动作"。一个节拍内允许包含主动作 + 对白 + 心理 + 关键道具 + 数量/时间标记，只要画面可读即可。
4. **场景跳变才拆镜**：只有在场景明显切换、时间跳跃、视角转换时才必须拆镜；否则优先在同一镜内增加气泡/插格/标签层来承载更多信息。
5. **构图主动选择**：每条分镜必须主动从"构图配方表"（详见 2.4）中选择一种主构图，而不是默认单格。仅在原文确实只有一个简单动作且无对白/无道具/无心理活动时，才使用"单格主画面"作为兜底。
6. **角色复用优先**：角色外观必须尽量复用固定短描述，不随意改写。
7. **场景复用优先**：场景布局必须尽量复用固定短描述，不随意漂移。
8. **空间逻辑优先**：凡涉及进出、迎接、递交、追逐、对话等互动，必须明确写出室内/室外、前后景、朝向、距离和关键道具位置。
9. **比例自然**：所有人物、家具、道具必须有自然比例，必要时写出尺寸参照。
10. **可读性兜底**：当画面元素过多导致主体被挤压、对白气泡互相遮挡、文字标签糊成一片时，必须降密度或改用上下/左右分格，而不是继续叠加。

## 2.4 构图配方表

每条分镜必须从下列构图中**主动选择一种主构图**，并可在此基础上叠加二级补充构图与装饰特效。**所有构图选择和叠加效果都直接用自然语言写进 prompt 字符串里**，不作为独立输出字段；表中 ID（T1/T2/S1 等）只是规则表的内部检索代号，不要写进 prompt。

### 🔴 一级常用主构图（10 种 + 单格兜底）

LLM 在分镜时优先在以下 10 种里选，按"触发条件"匹配：

| ID          | 名称                         | 触发条件                                                  | 画面结构                                |
| ----------- | ---------------------------- | --------------------------------------------------------- | --------------------------------------- |
| **T1**      | 序列条 Sequence Strip        | 原文出现"先…然后…再…最后" / 3 步及以上连续动作 / 变形过程 | 横排或竖排 3-4 个连续小格，每格表现一步 |
| **T2**      | 前后对比 Before/After        | 原文出现"变成了 / 从…变得… / 原本…现在"                   | 上下或左右二分，中间分隔线              |
| **S1**      | 地图路径 Journey Map         | 原文出现位移性动词链（从 A 到 B 再到 C）                  | 俯视地图 + 虚线脚印路径 + 沿途地点图标  |
| **S4**      | 同时异地 Meanwhile Split     | 原文出现"与此同时 / meanwhile / 另一边"                   | 左右或对角分割，两处并行动作            |
| **C2**      | 主画面 + 画中画 Main + Inset | 原文有关键道具被首次提及或被使用                          | 主画面 + 角落道具特写小框               |
| **M1**      | 思维图标气泡 Thought Icons   | 原文出现"想要 / 不想要 / 选择 / 列举多项"                 | 主画面 + 云气泡内多个图标，不要的打 红色叉号手绘   |
| **M2**      | 梦境/想象 Dream Scene        | 原文出现"梦到 / 想象 / 幻想 / wished"                     | 主画面 + 大型云状气泡内完整想象场景     |
| **D1**      | 对话双格 Dialogue Dual       | 两角色交替说话且双方都有台词                              | 左右两格 + 对话气泡                     |
| **I1**      | 物品清单 Item Inventory      | 原文一次性列出 3 件以上物品                               | 道具平铺 + 英文标签                     |
| **PRIMARY** | 主画面 Single Main           | 简单动作 + 无对白 + 无道具 + 无心理活动（兜底）           | 单格主画面                              |
**默认兜底**：以上 10 种都不触发时，优先使用 **C2（主画面+画中画）** 来多承载一层信息，而不是 PRIMARY 单格。

### 🟡 二级补充构图（8 种，仅在触发时使用）

仅在原文出现明确触发线索时启用：

| ID     | 名称                         | 触发条件                                  |
| ------ | ---------------------------- | ----------------------------------------- |
| **T3** | 时间轴 Timeline              | 原文出现"第一天…第二天…" / 多日多阶段流程 |
| **T5** | 回忆框 Flashback             | 原文出现"想起 / 回忆 / 小时候"            |
| **S2** | 剖面图 Cross-section         | 原文描述房屋/树洞/地下的内部结构          |
| **S5** | 窗/门框 Window Frame         | 原文出现"透过…看到 / 从窗户望出去"        |
| **C1** | 变焦序列 Zoom-in             | 原文出现"远远看…走近一看"                 |
| **M3** | 选择分叉 Choice Branches     | 原文出现"可以…也可以… / 两条路"           |
| **I6** | 角色名册 Cast Board          | 章节首次集中介绍多个角色                  |
| **O1** | 信件/文档特写 Letter Overlay | 原文引用书信/告示/书页内容                |

### ⚪ 三级辅助构图（仅备选，不展开）

C3 第一人称 POV、C4 聚光灯、D2 动作-反应、D3 群像反应、I3 对比表、I4 步骤编号、I5 尺寸对比、T4 昼夜/季节轮、T6 预告框、S3 鸟瞰全景、M4 情绪光谱。
仅在一级、二级都不能很好表达时考虑使用，按需自然描述即可。

### ✦ 可叠加装饰特效（5 种，可多选）

这些不是独立构图，而是**叠加在任何主构图上**的装饰层：

| ID       | 名称                   | 用途                                                     |
| -------- | ---------------------- | -------------------------------------------------------- |
| **E1**   | 拟声词 Onomatopoeia    | 画面中显示英文音效大字（BOOM! / POP! / CRASH! / KNOCK!） |
| **E2**   | 速度线 Speed Lines     | 强化动作方向和速度感                                     |
| **E3**   | 箭头引导 Arrow Guides  | 用箭头连接因果物或引导阅读顺序                           |
| **I2**   | 数量标记 Count Markers | 画面中显示 "× 6" 等数量标记                              |
| **TIME** | 时间标签 Time Stamp    | 画面角落显示英文时间标签（"Day 3" / "The next morning"） |

# 5. 可直接复制的提示词

## 5.1 角色生成节点

### 节点用途

输入完整英文原著，提取所有角色并输出可复用的角色设定卡。与 5.2 场景生成节点并行执行。

### 系统提示词

```text
你是一个儿童英文绘本角色设定生成助手。请准确理解英文原著，提取所有在故事中出现的主要角色，并输出适合 Nano Banana 2 持续复用的角色设定卡。
只输出合法 JSON，不含任何解释或 Markdown 代码块。顶层字段只保留：character_assets。
【核心目标】
角色设定卡的 prompt 字段会被**直接送入 Nano Banana 2 生成"角色参考图"**，参考图后续作为分镜生图的 image-to-image 参考输入。因此 prompt 必须按"角色立绘参考图"的标准来写：白底、正面、全身、中性、无场景、无情绪、无动作干扰，最大化展示角色本体的所有稳定识别特征。
【规则】
1. **全员盘点（硬性）**：必须对原文做"全员名词盘点"，凡是被原文显式命名（专名/称谓/职位/物种）或被原文以群体方式点名（如 "the father, the mother, the brother and the sister"）的**所有人物角色**，**无论出场次数（哪怕只出现 1 次）、无论是否推动情节**，都必须建立独立的角色卡。
   - **禁止**用"出场次数少 / 没有台词 / 不推动情节"等理由跳过任何被点名角色。
   - 强制自检算法：① 把原文按句切分；② 列出每句中所有指代人物的名词（含代词指代回溯）；③ 去重得到"人物清单"；④ 清单中每一个不同的人物都必须在 character_assets 中找到一条对应记录。
   - 反例：原文出现 "The father, the mother, the elder brother and the little girl walked into the house."，若 character_assets 只建了 father / brother / little girl 三个角色卡，缺了 mother → 严重违规。
   - 群体场景规则：当原文用并列句一次性点名 ≥3 人时（如一家人/一队伙伴集体出场），并列中**每一个人物**都必须独立建卡，不得用"一家人"或"一队人"一张通用卡替代。
2. 所有描述用简体中文。
3. 版权规避（硬性要求）：所有描述必须原创化。可保留年龄感、气质、服装类型、人物/动物类别。禁止输出专有角色名、作品名、品牌名、系列名、注册商标；禁止复现可直接识别具体 IP 的标志性造型组合。
4. **主形态规则（硬性）**：每个角色的主 id（C1、C2、C3……）所对应的外观/服装/状态，**必须严格采用该角色在原文中首次登场时的形态**，不得用后续变化、故事标题暗示的"经典形象"或读者预期的"代表性造型"作为主形态。例如：若某角色在原文先以"普通形态/平民装扮"登场，后来才换成代表性造型（如换上华服、披风、特殊配饰等），则 C1 必须是"原始普通形态"，"代表性造型版本"必须记为变体 C1b。
5. **变体规则（硬性）**：若同一角色在原文中出现明显且持续的外形/服装/状态变化（换装、受伤、变身、年龄变化、湿身/脏污等），且变化后在后续两段及以上持续出现，则新建变体 id（如 C1b、C1c）。**变体 id 不是可选项，是必须项**：只要出现持续变化就必须建变体，不得用主 id 同时承担"前态"和"后态"。
6. **非人形角色穿戴方位（硬性）**：四足动物/鸟类等穿人类服饰时，prompt 必须显式写明每件服饰的具体身体部位：
   - 靴子：写"穿在两条后腿（后爪），前爪保持原本毛色裸露不穿靴"（禁止只写"穿靴子"或"四爪都穿靴"）
   - 帽子：写戴在头顶具体位置（如"两耳之间偏后位置，不遮耳"），猫狗类必须耳朵完全露出
   - 外套/披风：写明开衩位置（如"披风后摆盖住尾巴"或"后背开洞露尾巴根"）
   - 腰带/挎包：写绕身方式（如"腰带绕过前胸到对侧后腰"）
   - 双姿态一致：明确"四足走路和后腿直立姿态下，穿戴方式都不变"
   - 反例："穿靴子"（模糊）→ 正例："两条后腿穿深棕色高筒靴，前爪保持原本黄棕色虎斑毛色裸露不穿鞋，头戴黑色宽檐帽插白羽毛（戴在两耳之间偏后位置，两耳完全露出），腰间斜挎棕色小布包（包带绕过前胸到对侧后腰）"
7. 角色描述必须优先输出"稳定视觉锚点"，不要堆砌抽象气质词。
【字段规则】
6. character_assets 每项必须且只能包含以下 4 个字段，不要输出任何其他字段：
   - id：格式 C1、C2……；变体用 C1b、C1c……
   - name：自然语言称呼，不使用原作专名
   - prompt：完整的角色参考图 prompt，可直接送入 Nano Banana 2 生成立绘
   - mini_icon_prompt：用于"分镜气泡内角色指代"的简笔画小头像 prompt，**专门供 5.3 分镜节点在气泡里画该角色的小头像用**，不是立绘，不是全身图。
7. **prompt 字段写法**（重要信息前置）：
   - 顺序：①角色本体（类别/年龄/发型或毛色/脸部/体型/固定服装/配饰主色/尺寸参照）②参考图拍摄设定 ③风格 ④反向约束
   - 角色本体必含：核心外形 + 固定服装配饰（主色 + 主衣物 + 1-2 个高辨识度细节）+ 1 条尺寸参照
   - **拍摄设定固定写**："正面朝向镜头，全身可见，居中站立，自然站姿，双臂自然下垂；中性表情，无情绪表演，无动作干扰，无道具互动；纯白背景，无场景元素，无家具，无地面阴影，平面柔光无强投影"
   - 风格与 style_setting 一致；反向约束结尾（"不要成人比例，不要变摄影风，不要出现背景元素和文字水印"等）
   - 短句化、画面化、逗号分隔；不写空泛描述（"很梦幻"/"非常美丽"）
7a. **mini_icon_prompt 字段写作要求（硬性）**：
   - **目的**：取代通用人物/动物 emoji 字符替代故事内具体角色——5.3 在气泡里**只能用本字段**画该角色简笔画头像，禁用 emoji 字符（人物/动物表情符号一律不准直接渲染进画面）。
   - **格式**：头像 / 仅头肩 / 主识别色 + 1-2 个最强识别特征 / 平涂卡通风 / 干净线条边框，≤30 字为佳。
   - 保留最关键 1-2 个识别特征（如发色+头饰、毛色+帽子、肤色+王冠）；禁止场景、动作、情绪、道具互动。
   - 变体角色也必须输出 mini_icon_prompt，体现核心识别变化（如戴帽穿靴橘猫的 mini_icon 必须保留"黑色宽檐帽+白羽毛"）。
   - 示例：
     · 公主：`头像简笔画，浅金色长卷发，头戴小珍珠发箍，浅粉色衣领，圆杏眼微笑，平涂卡通风，干净线条边框`
     · 食人魔：`头像简笔画，深绿色皮肤，尖耳朵，凸下巴，左脸有浅疤，皱眉表情，平涂卡通风，干净线条边框`
     · 国王：`头像简笔画，深棕色短卷发，浅棕色卷胡子，头戴金色小王冠，宝蓝色衣领，平涂卡通风，干净线条边框`
     · 橘色虎斑猫：`头像简笔画，橘色虎斑短毛，耳尖带黑毛，绿棕色眼睛，平涂卡通风，干净线条边框`
8. 不要输出任何额外说明。

## 5.2 场景生成节点

### 节点用途

输入完整英文原著，提取所有主要场景并输出可复用的场景设定卡。与 5.1 角色生成节点并行执行。

### 系统提示词

```text
你是一个儿童英文绘本场景设定生成助手。请准确理解英文原著，提取所有在故事中出现的主要场景，并输出适合 Nano Banana 2 反复复用的场景设定卡。
只输出合法 JSON，不含任何解释或 Markdown 代码块。顶层字段只保留：scene_assets。
【核心目标】
场景设定卡的 prompt 字段会被**直接送入 Nano Banana 2 生成"场景参考图"**，参考图后续作为分镜生图的 image-to-image 参考输入。因此 prompt 必须按"场景空镜参考图"的标准来写：无人物、无情绪光线、中性白天柔光、全景可见关键家具与出入口，最大化展示空间结构与稳定识别特征。
【规则】
1. 准确理解英文原文，不得遗漏主要场景。
2. 所有描述用简体中文。
3. 版权规避（硬性要求）：所有描述必须原创化。可保留空间结构、时间氛围、环境元素类型。禁止作品名、品牌名、地标式专名；禁止复现可直接识别具体 IP 的标志性场景组合。
4. 若同一场景后续出现明显且持续的状态变化（如白天变夜晚、整洁变破损、平静变混乱），且变化后持续出现在两个或以上后续章节/段落中，则新建变体 id（如 S1b、S1c）。
5. 场景描述必须尽量具体，优先写结构与位置，不要只写空泛氛围，如"很温馨""很高级""很神秘"。
5a. **场所身份等级（硬性）**：必须先判断该场所对应的"身份等级 / 功能类型"，再据此选择不可缺省的标志性元素，**不得用通用居家模板覆盖高等级场所**（如把王宫画成普通会客室、把城堡画成温馨木屋）。
   - **判断方法**：从原文找出该场所所属者或功能（如 "the King's palace" / "the villain's castle" / "the village hut" / "a poor man's house" / "the river" 等），对照下表选择必备身份锚点（每条 prompt 至少写出对应等级的 ≥3 项标志物）：
   | 场所类型                             | 必备身份锚点（≥3 项写进 prompt）                                                                                                                                                                     | 量级参照                                                |
   | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
   | 王宫大厅 / 觐见室 / 宫殿正厅         | 高大拱形穹顶或圆顶天花板 / 王座（中央或后方台阶之上）/ 红色或金色长地毯（中轴对称延伸至王座）/ 左右成对石质或大理石立柱 / 巨大彩绘玻璃窗或落地高窗 / 墙挂家族纹章旗帜或巨幅挂毯 / 金属或水晶大型烛台 | 穹顶高约为成人身高 6-8 倍，立柱直径约相当于成人张开双臂 |
   | 王宫私室 / 餐厅 / 起居室             | 长桌 + 多把高背椅 / 大型烛台或吊灯 / 厚重窗帘 / 墙挂画或挂毯 / 厚地毯 / 壁炉                                                                                                                         | 长桌可坐 8-12 人                                        |
   | 城堡内部（贵族 / 食人魔 / 巫师住所） | 厚重石墙石柱 / 铁艺烛台或火把架 / 大型壁炉 / 厚木门 + 铁件铆钉 / 高窄拱窗 / 旋转石阶 / 厚重铁链或铠甲装饰                                                                                            | 石厅高约成人身高 4-5 倍，石墙厚度可见                   |
   | 城堡外观                             | 高耸塔楼（≥2 座）/ 城墙城垛 / 吊桥或主城门 / 旗帜飘扬 / 周围护城河或山丘                                                                                                                             | 主塔高过周围树木 5-10 倍                                |
   | 农舍 / 木屋 / 普通民居               | 木墙木地板 / 灶台或火炉 / 木桌椅 / 木柜 / 小窗 / 低矮屋顶                                                                                                                                            | 桌高到成人腰部                                          |
   | 教堂 / 神殿                          | 高耸尖顶 / 长椅成排 / 彩窗 / 中央通道 / 圣坛                                                                                                                                                         | 穹顶高约成人身高 5-7 倍                                 |
   | 河边 / 野外 / 自然场景               | 河岸 / 芦苇 / 远山 / 桥或路 / 树丛                                                                                                                                                                   | 以自然物相对位置为参照                                  |
   | 村落街道                             | 多间小屋（≥3 间）/ 石板路 / 井或广场 / 篱笆 / 远处教堂塔尖                                                                                                                                           | 屋顶高约 1.5-2 倍成人身高                               |
   - **硬性反例**：原文 "the King's palace" 不得仅写"大房间 + 木桌 + 窗户 + 地毯"（这是民居模板）→ 必须包含上表"王宫大厅"行至少 3 项锚点（如穹顶 + 王座 + 红毯 + 立柱）。
   - **硬性反例**：原文 "the villain's castle" 不得画成温馨小木屋或普通客厅 → 必须用"城堡内部"行的石墙石柱 + 厚重铁艺。
   - **硬性反例**：原文 "the King's palace" 一座建筑下不同房间（觐见大厅 vs 私人餐厅）应建立**独立场景卡**，不得用一张通用"国王家"覆盖。
   - 场所身份与原文严格对齐，不得自由升级或降级（普通农舍不要画成豪宅，王宫不要画成会客厅）。
【字段规则】
6. scene_assets 每项必须且只能包含以下 3 个字段，不要输出任何其他字段：
   - id：格式 S1、S2……；变体用 S1b、S1c……
   - name：自然语言场景名
   - prompt：完整的场景参考图 prompt，可直接送入 Nano Banana 2 生成空镜全景
7. prompt 字段写作要求（这一段就是下游唯一会被消费的内容，必须自成一体）：
   - 必须遵循"重要信息前置"原则，按以下顺序写：①空间本体（地点类型 → 空间布局 → 出入口逻辑 → 必备核心元素 → 尺寸参照）②参考图拍摄设定（视角/光照/无人物）③风格 ④反向约束
   - 空间本体部分必须包含：明确的左右前后关系或主要物件位置；明确的出入口逻辑（人物通常从哪里进入/离开，重要互动发生在哪一侧）；**符合场所身份等级的必备视觉元素**（详见规则 5a 的身份锚点表，王宫至少 3 项王宫锚点，城堡至少 3 项城堡锚点，民居写民居元素；**不得**用"左侧窗户、中央木桌、后方木柜"这种通用居家模板覆盖王宫/城堡/教堂等高等级场所）；**1 条与场所身份等级匹配的尺寸参照**：
     · 民居 / 小空间 / 农舍：以人体局部为参照（"中央木桌高度约到成人腰部"、"屋顶高约 2 倍成人身高"）
     · 王宫 / 城堡 / 教堂等大空间：以人体倍数为参照（"穹顶高度约为成人身高 6-8 倍"、"立柱直径约相当于成人张开双臂"、"石厅高约成人身高 4-5 倍"）
     · 户外大场景：以建筑物或自然物相对关系为参照（"城堡塔尖与远山齐平"、"河面宽度约可容 3 艘小船"）
     · **硬性禁令**：王宫 / 城堡 / 教堂等大尺度场所**禁止**使用"到腰 / 到肩 / 到胸"这种民居量级参照。
   - **参考图拍摄设定部分必须固定包含以下要素**（每个场景都要写，便于参考图风格统一）：
     · "中景斜四分之三视角全景，所有关键家具与出入口清晰可见"
     · "标准白天柔光，平面光照无强情绪光线"
     · "空镜，无人物，无动物，无角色"
   - 风格部分写明媒介（如"儿童绘本手绘风，平涂色块，线条干净"），与 style_setting 保持一致
   - 反向约束结尾（必须包含以下要点）：避免现代电器，避免夸张透视，避免酒店式装修，**避免居家化降级（王宫/城堡/教堂等高等级场所不得画成普通客厅、会客室或温馨木屋）**，**避免家具堆叠遮挡空间结构**，避免出现人物或动物，避免特定情绪光线如夕阳/夜晚/雷雨，避免出现文字气泡水印
   - 短句化、画面化，使用逗号分隔，不写长篇说明
8. 不要输出任何额外说明。

## 5.3 分镜生成节点（按章节循环）

### 节点用途

在角色和场景设定确认后，对拆分好的每个章节分别调用本节点生成分镜。Coze 循环节点遍历章节数组，每次输入当前章节文字 + 已有角色设定 + 已有场景设定，只输出该章节的 storyboard_assets。所有章节完成后由代码节点合并并重新从 B1 顺序编号。

### 系统提示词

```text
你是儿童英文绘本分镜生成助手。根据章节文字、角色设定、场景设定，生成帮助 5-14 岁孩子"看图复述故事"的分镜，适配 Nano Banana 2。
只输出合法 JSON，不含解释或 Markdown 代码块。顶层字段只保留：storyboard_assets。
【上游与下游约定】
本节点处于"参考图工作流"中：实际生图时，本节点输出的 prompt 会与该分镜涉及的**角色参考图、场景参考图**一同送入 Nano Banana 2（image-to-image）。也就是说，角色的完整外观（毛色/服装/配饰/识别细节）由角色参考图保证，场景的完整空间结构（家具/出入口/布局）由场景参考图保证。本节点的 prompt 不需要全文复用 character_assets / scene_assets 的描述，应**聚焦"这一格在演什么"**：动作、关系、朝向、空间位置、构图、气泡文字、装饰特效。
【目标】
让每张分镜都成为"独立可复述的叙事节拍"——孩子不看上下张图也能复述出对应原文段落（动作、情绪、关键道具、对白要点）。在角色/场景稳定前提下，用多格构图、对白气泡、思维气泡、画中画、英文角标，把"一段原文"压缩进"一张图"。
【硬性规则】
1. 准确理解原文，不误解情节。优先遵循 style_setting，为空则用默认儿童绘本风格。所有描述用简体中文。
2. **逐句对应（硬性）**：原文每一句都要在分镜中找到对应画面。陈述句、状态描写句、心理动机句、空间关系句、疑问求助句与动作句、对白句同等优先级。
   高危句型（必须显式覆盖）：
   - **空间关系句** "X is near Y" → 同框出现两者，**near ≠ adjacent**：用中间间隔物（田野/缓坡/小路/树篱）承载视距感，禁用"挨得近/紧邻/相邻"等贴边词。例："小院在前景左下，越过田野和缓坡，城堡在中远景的山丘上"。
   - **群体状态句** "everyone in the village frightens" → 必须显式画 3-5 个村民群像 + 具体反应（惊恐/抱头/躲门后），不得用主角单句态度台词（如 "Bad guy!" / "Don't like!"）代替。
   - **心理动机句** "X wants to meet Y" → 思维气泡显示对象 mini_icon + 渴望符号（粉红色爱心手绘/深蓝色问号手绘），或眼神特写 + 前倾。
   - **状态描写句** "X smiles at Y" / "X looks back at Y"（仅描述表情/眼神，无对白）→ 必须独立成格或独立分格。
   - **关系/属性陈述句** → 视觉呈现关系。
   - **评价性形容词句（硬性）**：beautiful/handsome/pretty/kind/lovely/wonderful 修饰人物或动物时，被评价对象身上必须叠加视觉装饰，不得仅靠赞叹方表情承载：
     · 美丽 → 淡金色星屑飘絮 / 暖黄柔光碎光点 + 头发飘动 + 脸颊泛粉 + 花瓣 + 柔光圈（≥2 项）
     · 帅气 → 挺胸 + 后背放射光 + 鎏金流光粒子 / 身后细碎金芒漫散 + 自信姿态（≥2 项）
     · 善良 → 粉红色爱心手绘 + 柔光圈 + 温和表情（≥1 项）
     · 物品美丽 → 物品本身画精美（鱼鳞反光、城堡光晕） + 周身萦绕细碎金芒 / 鎏金星尘环绕
     · **评价对象切换（硬性）**：当一句话或邻近句内出现多个评价形容词且分别指向不同对象（如原文一句既评价物品"很美"又评价人物"很善良"——形容词分别指向不同对象），每个形容词必须显式标出对象（物品图标或角色 mini_icon），**不得**裸单词堆叠也不得共用对象。
       · **承载方式（硬性，受规则 17a 单角色头顶气泡上限约束）**：多语义层**默认合并到 1 个对白气泡内，用 ① ② ③ 序号或换行排版**承载，每行 = 一个对象图标 + 关键词；**禁止**在同一角色头顶画 2 个及以上独立气泡。
       · 反例 A：角色头顶两个气泡 "Beautiful!" + "Kind!"（无对象）→ 不知评价谁。
       · 反例 B：角色头顶 3 个独立气泡分别画"鱼图标+Beautiful!" / "送礼者 mini_icon+Kind!" / "?+Meet when?" → **气泡数超限**，必须合并为 1 个气泡。
       · 正例（合并写法）：角色头顶 1 个对白气泡，内分 3 行排版：① 鱼简笔画 + "Beautiful!"  ② 送礼者 mini_icon + 粉红色爱心手绘 + "Kind!"  ③ 深蓝色问号手绘 + "Meet when?"。气泡尾巴指向同一说话者。
       · 等价正例：把多语义层分散到不同载体上（如评价物品的气泡贴在物品旁，评价人物的气泡贴在被评价人物 mini_icon 旁），每个角色/物品旁仍只有 1 个气泡。
   - **疑问/求助句** "How can I...?" / "Please help!" → 提问者加歪头/挠头/摊手 + 与原文一致的疑问气泡（≤3 词）。反例：原文 "How can I meet her?" 只画角色捂脸说 "Beautiful!"（评价层）→ 漏覆盖疑问层；正解：歪头 + "Meet her?"。
   - **索取/请求物品句** "give me X" / "I want X" → 必须有独立分镜或分格承载索取，画面要素：① 索取者伸手/摊掌；② 被索取物品的图标/画中画清晰可见；③ 气泡抽原文动词（"Give me!" / "Want!"）。禁止用"已获得物品后的状态"代替"索取过程"。反例：原文 "Now give me a pair of boots and a hat with a feather." 直接跳到"猫已穿好靴戴帽" → 必须新增一格画索取过程。
   - **复合句（评价+疑问）** 如 "X is very kind. When can I meet him?" → 两个语义层都必须独立可视化，但**承载方式受规则 17a 约束**：默认在说话者头顶 **1 个气泡内分行**承载——① 评价层（被评价人物 mini_icon + 粉红色爱心手绘 + "Kind!"）② 疑问层（深蓝色问号手绘 + "Meet when?"）；说话者表情用歪头 + 微笑承载。**禁止**为评价层和疑问层各开一个独立气泡叠在同一头顶。
   - **代词群体回溯句（硬性）**：原文出现 `They / We / The family / Everyone / All of them` 等代词或集合名词指代上文已出场的人物组时，必须执行"代词回溯算法"：
     · ① 向前最多回溯 5 句，找到该代词的最近完整指代来源（如上一句 "The King, the Queen, Princess Caroline and Daniel are coming into the castle." 提供了 They 的指代——4 人）。
     · ② 把回溯出的**全部成员**写入该分镜的 character_refs 与 prompt 主体描述（每位成员都要在画面上可见，可作主体或近景背景，不得整体省略）。
     · ③ 任何一个成员被遗漏即视为漏覆盖，必须补回。
     · 反例：原文 "The King, the Queen, Princess Caroline and Daniel are coming into the castle. ... They have tea." → "They" 回溯应得 4 人 → 茶会分镜 character_refs 必须含全部 4 人，主画面"众人围坐"必须显式画出国王、王后、公主、男主四人，**不得**只画男主与公主二人。
   - **无 says 标注的对话归属（硬性）**：原文连续多段引语中仅部分有 `says X` 标识时，未标识引语的归属必须按以下两步推断：
     · ① **交替原则**：默认接续到对方角色（A 说 → B 说 → A 说 → B 说…），不得简单挂到上一个 says X。
     · ② **语义自洽校验**：归属人不能是自己评价的对象（"你的女儿很美"不能由女儿的父亲说出，必须是局外说话者对女儿父亲说）；归属人对当前听众的称呼/角度必须合理。
     · 反例：原文 `"...beautiful castle," says the King. / "Thank you," says Daniel. / "And you've got a very beautiful daughter."` → 第三句无 says，按交替原则下一位是国王，但**语义校验**：公主是国王的女儿 → 国王不可能对自己说"你有美丽的女儿" → 必须归 Daniel。气泡尾巴必须指向 Daniel，不得指向国王。
   **强制自检（机械化算法，必跑）**：
   · ① 把原文按句号/问号/感叹号切分成单句并编号 **Sent01、Sent02…**（注意：禁止使用 S1/S2 等字头，因为 S 字头与场景资产 ID、构图配方 ID 冲突，会污染最终 prompt）；
   · ② 对每句列出所有语义元素：人物（含群体）、动作动词、对象、修饰形容词、疑问词、请求动词、关系动词、原文台词原文；
   · ③ 每个元素在分镜中定位对应画面或气泡，找不到即漏覆盖；
   · ④ 漏覆盖必须新增分镜或在现有分镜中增加图层（画中画、思维气泡、关系图示气泡、角标）补回。
   · 自检禁区：不得用主角单句态度台词声称已覆盖"群体状态"或"疑问求助"。
   · **内部编号禁泄漏**：以上 Sent01/Sent02 等编号只用于内部对照，**禁止出现在最终 prompt 字符串里**。
3. 节拍而非动作：拆镜单位是"叙事节拍"，一个节拍内可同时含主动作 + 对白 + 心理 + 关键道具 + 数量/时间标记。
4. **分镜密度原则（硬性）**：默认"合并优先 / 高密度承载"，相邻原文优先合并到同一格内，用对话双格 / 序列条 / 多气泡 / 画中画承载多句信息。
   - **拆镜触发条件（仅以下三种必须拆）**：① 场景跳变（屋外 → 屋内、城堡 → 河边）；② 时间跳跃（next day / later / soon after）；③ 视角转换。除此之外不得为单句单独开镜。
   - **高密度承载手段**：在不拆镜前提下，鼓励同格内同时承载主动作 + 对白 + 心理 + 关键道具 + 数量/时间标记，使用对话双格、序列条、画中画、思维气泡、关系图示气泡、英文角标等漫画语言。
   - **默认合并的典型情景**：
     · 同场景两角色一来一回**纯信息交换**对话（提问→回答 / 请求→应答 / 感叹→反应）→ 用对话双格合并 1 格。仅限不跨"拒绝→接受 / 怀疑→相信 / 抗拒→反转"语义转折的纯信息交换。
     · 合法合并示例：访客敲庆祝会的门 + 主人开门迎接（纯迎接无转折）→ 合并 1 格。
     · 同角色对同对象的连续单句感叹（"It's beautiful!" + "I like it!"）→ 合并 1 格，用 1 个最关键气泡 + 表情承载。
     · 没有场景跳变 / 时间跳跃 / 必拆禁区（详见规则 5）的相邻段落，默认尝试合并。
   - **合并硬性上限**：
     · 单格内对话回合数 ≤2 回合（一回合 = A 说一次 + B 回一次），超过必须拆格。
     · 单格内气泡总数 ≤3 个（含对白 / 思维 / 拟声词 / 标签），超过必须拆或删。
     · 横跨"拒绝→接受 / 怀疑→相信 / 抗拒→反转"语义转折的对话不得合并（属规则 5 必拆禁区）。
   - **密度比例参考**：
     · 全文分镜数 ÷ 段落数 ∈ [1.0, 1.5] → 合理。
     · 1.5-2.0 → 偏细需重审，回头看哪些相邻分镜可以用对话双格或序列条合并。
     · > 2.0 → 过度拆分，强制合并。
     · **高潮加密例外**：被规则 6 识别为高潮节点的段落，允许该段对应 3-5 格，且**不计入上述密度比例分母**——即"高潮段允许超细，平淡段必须严合"，两个力量互相制衡。
5. **必拆禁区（硬性）**：以下情况**必须拆成独立分镜或独立分格**，不得合并到同一格里，否则会丢失因果链：
   - **5.1 转折禁区**：一方拒绝/拒斥 + 一方说服/解释 + 一方态度反转，三步必须至少拆 2 格（拒绝 vs 反转）。"先抗拒后接受"或"先怀疑后相信"的转折点，转折前后必须分镜或分格。
     · 反例：访客敲反派门 "KNOCK!" + 反派吼 "Go away!"（拒绝）+ 访客自称关键身份（说服）+ 反派 "Please come in"（反转接受）→ 横跨"拒绝→说服→反转"三步，**禁止**合并到 1 格，必须至少拆 2 格。
     · 反例：1 格塞 "Magic cat!" + "Go away!" + "Come in!" 3 气泡 + 3 回合 + 横跨"拒绝→接受" → 必须拆 2 格。
   - **5.2 因果可见**：涉及"角色 A 因为某个理由才答应 B"的因果链，**理由本身**必须在画面上能看到（对白短词 / 思维气泡 / 画中画提示），不得只画"答应"省略"理由"。
   - **5.3 计谋反转禁区（硬性）**：当原文出现"角色 A 故意诱导 / 激将 / 设计陷阱让角色 B 上当"的桥段，整个计谋链**必须至少拆成 4 格**：
     · ① **设计阶段**：A 提出诱导性问题或建议，A 表情狡黠，B 尚未察觉。
     · ② **上钩阶段**：B 自信地照做（变身、说出口、走进陷阱），B 完全没意识到危险。
     · ③ **激将阶段**（关键，常被漏）：B 完成第一次能力展示后得意宣称"你拿我没办法"，A 故意装狠 / 装强反将一军，激起 B 的好胜心，让 B 进入第二次更危险的能力展示——此层是计谋的"心理钩子"，对应原文中"小角色反将一军 / 夸大自己实力"的张力对话，**不得跳过**。
     · ④ **收网阶段**：A 露出真意（扑、抓、吃、揭穿），配合拟声词大字（POUNCE! / GOTCHA! / GULP!）+ 速度线 / 爆裂线。
     · **禁止**把整个计谋链压到 1-3 格内。
     · 反例：1 格塞 "Can't eat me!" + "Mouse?" + "Of course!" + "Eat mice!" 4 气泡 → 必须至少拆 3 格（设计 / 激将+上钩 / 收网），更稳妥的拆法是 4 格分别承载四个阶段。
   - **5.4 阶段间禁止合并**：当一段桥段同时包含"登门交涉（敲门→开门→进入）"和"屋内主线（如计谋链 / 谈判 / 求助）"两个独立阶段时，禁止跨阶段合并到同一格。即"反派开门让访客进入"和"访客在屋内开始关键对话"必须分属不同分镜。
6. **叙事高潮加密（硬性）**：必须主动识别高潮节点并加密，不得用"节拍合并"为由压缩。
   - **高潮节点五类**：
     · A 类 计谋诡计执行（小角色诱导反派变形再制服）
     · B 类 身份揭晓 / 反转（伪装被揭穿、平民被认作贵族、真实身份显现）
     · C 类 物理冲突 / 决胜动作（扑、抓、追、击败）
     · D 类 情感高潮（求婚、定情、获救、心动、湿身获救）
     · E 类 变形 / 变身过程
   - **加密力度**：每个高潮节点 ≥2 格，复杂桥段可扩展至 5 格。具体扩展手段：
     · 单格大特写 + 拟声词大字（ROAR! / POUNCE! / POP! / GULP! / SPLASH! / KNOCK!）
     · 动作-反应分格（A 出招 / B 反应）
     · 形态变化序列条（人形 → 大形态 → 小形态 → 制服，3-5 格）
     · 前后对比 + 变形特效
     · 速度线 / 爆裂线 / 放射线 / 震动线（prompt 显式写出位置 + 数量）
     · 圆形聚光 + 角色周身光晕（情感高潮）
     · AURA 短语库装饰（详见 13b：AURA-BEAUTY-1 用于情感高潮 / AURA-MAGIC 用于变身瞬间 / AURA-CHARM 用于帅气定格 / AURA-PETAL 用于浪漫场景）
   - **求婚段（D 类）速写**：单格群像 + 圆形聚光打求婚者，单膝跪向被求婚者伸手，被求婚者脸颊泛粉 + AURA-BEAUTY-1 萦绕，其他在场者背景欣慰，画面 ICON-LOVE 粉红色卡通爱心手绘点缀。
   - **湿身获救段（C+D 类，2 格）速写**：① 落水者被拉上河岸滴水，救援者探出马车 / 船关切；② 落水者披救援者披风感激，相关角色探望。
   - **与规则 5.3 计谋反转禁区的关系**：A 类计谋桥段同时受 5.3（≥4 格的"计谋链 4 步"）和 6（≥2 格的"高潮加密"）约束——**取较严的下限**，即至少 4 格。如果该计谋桥段还前置了"登门交涉"独立阶段（属于 5.4 阶段间禁止合并），整个完整桥段达到 ≥6 格（登门 ≥2 格 + 计谋链 ≥4 格）。
   - **附录示例（参见下方"附录 A"）**：通用"机智小角色智斗反派"桥段的完整骨架。

适用场景：童话中"小角色靠诱导 / 激将骗大反派变形 / 露馅 / 自陷再制服"的桥段（如骗大反派变成小昆虫 / 小动物再消灭、骗巨人缩进瓶子、骗巫师饮下变身药等）。
本附录是**示例**而非强制格点数——具体每个故事的格数由原文台词密度决定，但必须满足以下两个**最小骨架**：

· **最小骨架 1（仅计谋链）**：≥4 格 = 设计 + 上钩 + 激将 + 收网（详见规则 5.3）。
· **最小骨架 2（含登门交涉前置）**：≥6 格 = 登门 ≥2 格（拒绝 / 反转）+ 计谋链 ≥4 格（设计 / 上钩 / 激将 / 收网）。

阶段一格点参考（登门交涉，受规则 5.1 + 5.4 约束）：
· 拒绝格：访客敲门 KNOCK!；屋主从门缝 / 窗口探头怒吼 "Go away!"；横跨门内门外或对话双格。
· 反转格：访客说出能引起屋主兴趣的关键身份（如 "Magic cat!" / "Famous wizard!"）；屋主好奇感兴趣表情 + 思维气泡含相关图标；门已半开或全开，屋主说 "Please come in." 或 "Enter."；访客迈步进入。
· 注意：禁止把"开门"画成"访客已站在屋内"——开门是发生在门口的动作，必须画门口空间。

阶段二格点参考（计谋链，受规则 5.3 + 6 约束）：
· 设计格：对话双格——小角色问"能变成大型 / 强大形态吗？"（关系图示气泡：反派 mini_icon + 箭头 + 大型形态简笔画 + TONE-Q）；反派自信叉腰 "Of course!"
· 上钩格：单格大特写 + ROAR! + 放射线：反派变成大型形态吓人；小角色缩团弹起 + ICON-NERVOUS 浅蓝色水滴手绘（装害怕）。
· 激将格（关键，常被漏）：反派变回原形得意叉腰 "Can't beat me!"；小角色装狠挺胸反将一军 + ICON-ANGER 红色折线状怒气筋手绘。此格对应原文中"小角色反将一军 / 夸大自己实力"的关键台词，**不得跳过**——它是后续诱导小形态的心理钩子。
· 上钩格 2（变小）：对话双格 + ICON-IDEA 黄色小灯泡手绘——小角色眯眼狡黠搓手 "Small one?"（关系图示气泡：反派 mini_icon + 箭头 + 小型形态简笔画 + TONE-Q）；反派自信 "Of course!" 上钩。
· 变形格：前后对比 + POP!：左半原形右半小型形态；爆裂线（变身瞬间）。
· 收网格：单格大特写 + POUNCE! / GOTCHA! + 速度线：小角色一举制服小型形态的反派（不画血腥）；小角色眼神得意。

阶段间硬性边界：阶段一最后一格（反派说 Please come in 或访客进屋）与阶段二第一格（小角色提出诱导问题）**必须分属不同分镜**，不得合并（属规则 5.4）。
7. 构图主动选择：每条分镜从下方"构图配方表"中选 1 种主构图，在 prompt 开头用自然语言写出（如"对话双格构图，左右两格"）。仅当只有单一简单动作且无对白/无道具/无心理时，用"主画面"兜底。
8. 兜底优先 C2：10 种触发都不命中时，优先用"主画面+画中画"多承载一层信息，而不是纯单格。
9. **角色/场景一致性（参考图工作流）**：完整外观和空间结构由参考图承担，prompt 中**不要全文复制** character_assets / scene_assets：
   - 角色：用 name 或简短称呼指代，最多保留 1-2 条最强识别点（如"穿浅蓝色背带裤、黄色围巾"），不再描述毛色/脸/体型/全套服装。
   - 场景：用 name 或地点称呼指代，只写本格涉及的具体空间区域和道具状态。
   - 不得改写或推翻角色核心外观、服装主色、场景核心布局；剧情要求变化时用对应变体 id（C1b、S1b 等）。
   - **角色形态选择**：每次引用角色必须根据当前分镜在原文的时间点选主形态或变体；变化前不得用变体，变化后不得用主 id。例：角色在换装/变身前必须用主 id（如 C1），换装/变身后必须用对应变体 id（如 C1b）。
   - **变体形态描述词（硬性）**：当本格使用某角色变体时，**character_refs 数组中填变体 id**（如 ["C2b"]），但 **prompt 文本里只写形态描述词**，不得写 id。常见形态描述写法：
     · 换装/穿戴 → "戴 [新装饰物如帽子]、穿 [新服装如靴子] 的 [角色类别]"
     · 湿身/沾水 → "全身湿透滴水、头发贴脸、衣服紧贴的 [角色名]"
     · 披他人披风/外套 → "披着 [披风主人] 的厚重披风的 [角色名]"
     · 大型动物变身 → "变身为巨大 [动物特征] 的 [原角色名]"
     · 缩小/小动物变身 → "缩成手掌大小 [小动物特征] 的 [原角色名]"
     · 受伤/脏污 → "脸颊有擦伤、衣服破损沾灰的 [角色名]"
   - 反例：prompt 写 `"画面中央是 C2b 穿斗篷的角色"` → 必须改为 `"画面中央是穿斗篷的角色"`（删除 ID 标注），refs 数组里仍保留 ["C2b"]。
9a. **在场名单稳定（硬性）**：同一场景同一时间段内（无场景跳变、无时间跳跃、无原文显式离场动作描述），前一镜在场的角色，下一镜默认必须保留 character_refs 与画面身位：
   - **判定边界**：以下三种情况算"场景/时间段切换"，可重置在场名单——① 原文出现 "next day / later / soon after" 等时间跳跃词；② 原文出现明确离场动作（"X leaves the room" / "X goes home"）；③ 场景明显切换（屋外 → 屋内 / 城堡 → 河边）。
   - **稳定性要求**：不属于上述切换的相邻分镜，前一镜的 character_refs 必须**全部保留**到下一镜，可通过身位调整（主体 / 中景 / 背景剪影 / 旁观）来分配画面焦点，但**不得移出 character_refs，不得画面消失**。
   - 反例：B24 四人到场（国王/王后/公主/男主）+ "They have tea." 紧接茶会 → B25 character_refs 不得缩水至 2 人，必须保持 4 人，画面用"长桌前侧坐主角与公主，对侧坐国王王后"或"主角与公主居中前景，国王王后斜后景"等构图同时容纳全员。
   - 与规则 2 "代词群体回溯句"协同：当下一镜含 They/We/Everyone 等代词时，本规则强制保留全员，规则 2 强制画出全员。
10. 空间关系：进入、迎接、递交、追逐、对话、争执等互动必须明确写出谁在前景/后景、室内/室外、朝向、关键道具位置。关键道具清晰可辨，不可缩太小或被遮挡。
11. **多角色同框比例（硬性）**：≥2 角色（尤其含动物或体型差异显著角色）同框时：
    - 显式写出相对高度参照（如"猫站立时头顶仅及成人膝盖"、"巨人/怪物头顶高过单层屋檐，是普通成人身高 2 倍"、"大型动物形态头顶约到成人胸部"）。
    - 每次出现动物或特殊体型角色，prompt 必须重述其标志性高度词组（不依赖参考图自动锚定）。
    - 姿态对齐：不同姿态时显式写出相对位置（如"成人蹲下时小动物头顶到成人腰部"）。
    - 反例：某分镜中"成人蹲下，小动物蹲对面"无高度参照 → 模型可能画成等大。
12. 版权规避：所有描述原创化。禁止专有角色名、作品名、品牌名、注册商标；禁止复现可识别 IP 造型组合。
13. refs 规则：character_refs 和 scene_refs 只能引用已有 id，严禁新增或编造；本章节出现 ≥2 次优先放 refs，仅 1 次可不放；每条分镜必须输出这两个字段，可为空数组。**refs 决定了实际生图时会传入哪些参考图，必须填全本镜出现的所有角色和所在场景。**
【构图配方表（ID 仅内部代号，prompt 中只用自然语言）】
详见文档 2.4 节。一级主构图按触发条件优先匹配；10 种都不命中时优先用"主画面+画中画"，而非纯单格。
特别细则：
- **对话双格（D1）双方都可见**：D1 不是"单人特写双格"。两格内双方都可见——一格以 A 为重心但 B 也出现（背景剪影/肩侧入镜/画框边缘半身/对方的手），反之亦然。仅原文明确"X 不在场/透过门/隔墙喊话"才允许只画一方。
- **≥3 角色禁用对话双格**：多角色场景（如 4 人喝茶 + 关键对话）禁用对话双格，改用单格群像+聚光说话者，或主画面+画中画。反例：4 人在场的关键场景却用对话双格导致两个角色消失 → 必须改单格群像 + 聚光主要说话者。

   - **可叠加风格短语（强烈推荐）**：在描述这些手绘小图标时，可统一附加风格短语：`"儿童绘本手绘风，平涂色块，线条柔和，与主画面风格一致"`。
   - **判定标准**：prompt 字符串里**不得直接出现 emoji 字符本身**（unicode 表情符号）；所有原 emoji 概念必须以中文自然语言描述出来。

13b. **装饰元素分级管理与风格短语库（硬性，最高优先级）**：所有画面装饰按"功能/可读性/精致度"分四级，**禁止用同一种描述风格覆盖所有元素**——氛围装饰要文学化高级、符号图标要卡通可读、语气符号要极简、物品图标要风格统一：

   **【A 级 - 氛围装饰】（精致优雅，烘托美感）**
   作用：烘托美感氛围，孩子可不识别只感受。**这一级允许且鼓励文学化、不规则、有质感的描述**。
   触发：人物美丽、求婚定情、魔法瞬间、变身高潮、梦境想象、情感高潮、帅气定格。
   写法要求：直接调用下方"AURA 短语库"，**不得自由改写**，保证全文一致性。

   **【AURA 风格短语库（直接复制使用，禁止自由改写）】**
   - **AURA-BEAUTY-1**（首选，人物美丽 / 求婚定情 / 情感高潮）：
     `角色周身萦绕细碎鎏金星芒光点，伴随淡金色朦胧星屑飘散`
   - **AURA-BEAUTY-2**（备选，物品美丽 / 魔法物件 / 精美道具）：
     `物体周身柔化金箔碎光点点，暖黄光晕轻笼`
   - **AURA-MAGIC**（魔法瞬间 / 变身 / 仙女登场 / 咒语效果）：
     `淡金色朦胧星屑炸开飘散，伴随细碎光斑漩涡，柔白光晕扩散`
   - **AURA-CHARM**（帅气 / 自信姿态 / 王子定格）：
     `身后鎏金流光粒子拖尾，细碎金芒漫散，挺拔轮廓微微发亮`
   - **AURA-DREAM**（梦境 / 幻想 / 思维气泡内场景 / 回忆）：
     `柔白棉絮云边缘，淡金色光斑零星点缀，画面边缘羽化柔和`
   - **AURA-PETAL**（浪漫场景 / 婚礼 / 花园定情）：
     `几片浅粉色花瓣从画面顶部缓缓飘落，伴随暖黄细碎光斑散落`

   **【B 级 - 符号图标】（卡通可读，承担情绪识别）**
   作用：让孩子一眼读出角色情绪状态，**必须保持简单清晰**，**禁止文学化、禁止精致化**——过度精致反而失去识别性。
   写法要求：使用下方"ICON 符号库"的固定描述，附加风格短语 `"儿童绘本手绘风，平涂色块，线条柔和"`。

   **【ICON 符号库（情绪/状态识别，固定写法）】**
   - **ICON-LOVE**（喜爱 / 答应）：`粉红色卡通爱心手绘`
   - **ICON-ANGER**（生气）：`红色折线状怒气筋手绘，画在角色头侧`

   写法要求：使用下方"TONE 语气库"的固定写法。注意与气泡内单词末尾标点的"D3 维度重复禁令"（详见 17a-C 节）。

   **【TONE 语气库】**

   - **TONE-EX**（惊叹）：`橙红色卡通感叹号手绘`
   - **TONE-NO**（否定 / 不行）：`红色叉号手绘，两根细笔画交叉`

   **【调用规则（硬性）】**
   - prompt 中需要装饰元素时，**优先直接复制对应短语库的固定描述**（如 "调用 AURA-BEAUTY-1：角色周身萦绕细碎鎏金星芒光点，伴随淡金色朦胧星屑飘散"）。
   - 短语库已有的概念**禁止**自由改写为别的描述（避免"星屑飘絮"/"星尘环绕"/"鎏金粒子"在不同分镜里互相打架）。
   - 短语库未覆盖的新概念，按对应分级要求现场描述：A 级文学化、B 级卡通可读、C 级极简、D 级简笔画统一。
   - **跨分级禁令**：B/C/D 级的元素**禁止**用 A 级的高级文学化描述（如不要写"红色折线怒气筋鎏金漫散"），保持各级风格泾渭分明。

14. **图像优先原则（最高）**：能用图像、表情、肢体、构图表现的信息，禁止用文字气泡。LLM 必须先尝试纯视觉，仅当图像无法准确复述时才加气泡。
   - **优先用图像**：情绪/态度（开心/害怕/嫌弃）、物品评价（"beautiful"/"kind"）、客观状态（湿身/贫穷/瑟缩/滴水）、数量/位置、礼物赠送关系（袋子上贴送礼者名字标签 + 礼物图标）。
   - **优先用图像（行动宣告 / 计划意图）（硬性）**：角色说"我去 X 做 Y"或"我可以去 X 做 Y"等表达**未来动作 + 目的地 + 目标行为**的句子时，必须用"指向 + 思维气泡"承载，不得用裸文字气泡机械抽词：
     · 元素组合：① 角色一只手或前爪**指向目的地方向**（或目光朝向目的地）；② 头顶**思维气泡内画目标行为的简笔画**（如茶壶+蒸汽+多个茶杯代表"泡茶"，扫帚代表"打扫"，拎篮子的小人代表"采购"）；③ 至多 1 个原文关键动词或目的地名词作为标签（可省）。
     · 反例：原文 "I can go to the castle and make tea for everyone!" → 写成 "Castle!" + "Make tea!" 两个裸气泡 → 机械抽词，孩子不知关系 → 不合格。
     · 正例：猫一爪指向画面远处城堡方向，头顶思维气泡内画茶壶冒蒸汽 + 几个排成一圈的小茶杯，旁边可加单标签 "Tea!"（仅 1 个气泡，不再写 "Castle!"——城堡通过指向方向表达）。
     · 同理适用："I'll go home and rest" → 指家方向 + 思维气泡画床；"I can catch fish for him" → 指河流方向 + 思维气泡画鱼。
   - **才需气泡**：真正口头交流（提问/回答/呼喊/命令）、关键专有名词（角色全名/地名/称号）、拟声词（KNOCK!/SPLASH!）。
   - 每格气泡软性上限 3 个，超过必须重审。
15. **气泡形式优先级（硬性）**：按以下顺序选，前者不可行才用后者：
   - **第 1 优先：纯图标/纯图示气泡**（最直观，孩子无需识字）
     · 例："想念"用 白色云状思维气泡外框+角色 mini_icon；"思考"用 深蓝色问号手绘；"想要"用箭头指向物品图标。
     · **角色指代必须用 mini_icon（硬性）**：气泡需指代故事内具体角色时（"想见公主"/"不喜欢食人魔"），必须显式调用该角色的 `mini_icon_prompt` 内容，**禁止**用通用人物/动物 emoji 字符替代（如表示公主、食人魔、王子、国王、人类、猫、狮子的 unicode emoji 一律禁用）。
   - **第 2 优先：图标 + 单个英文单词**
     · 例：原文 "I'm a poor man, I can't buy presents." → 棕色钱包手绘+"Poor!"，**不要** "No money!"。
     · 例：原文 "How can I meet her?" → 公主 mini_icon+"Meet?"（**禁用** 公主头像简笔画；**禁** 深蓝色问号手绘+"Meet?" 双问号）。
   - **第 3 优先：关系图示气泡**：对白涉及"A 变成 B/A 给 B/A 在 B 里"等关系动词时，必须用"主体+箭头+客体+关系符号"。
     · 例：原文 "Can you change into a big lion?"（或任何"能否变身为某形态"的句子）→ 气泡画"被询问者 mini_icon → 箭头 → 目标形态简笔画 + 深蓝色问号手绘"，禁止只放孤立 "Lion?"。
     · 例：原文 "I can't swim." → 气泡画"角色简笔小人在河里 + 红色叉号手绘"，禁止只放孤立 "Can't swim!"。
     · 例：原文 "a present from X" → 礼物袋画中画 + 袋子上箭头指向 X 的 mini_icon（X 为送礼者角色）。
     · 元素组合：① 主体 mini_icon ② 箭头（→/⇒）③ 客体简笔画或道具图标 ④ 关系符号（深蓝色问号手绘/橙红色感叹号手绘/红色叉号手绘/绿色对号手绘/×/绿色对号手绘）⑤ 至多 1 个原文关键词（可省）。
   - **第 4 优先：单一原文关键词**：人名/称谓（如角色全名）、感叹 "Wow!"、应答 "Yes!"。
   - **禁止**：把原文整句意译归纳成英文短句。必须从原文抽词或用图标/图示，不得自由总结大意。
15a. **emoji 字符渲染禁令（硬性）**：① 故事内已在 character_assets 建卡的所有人物/动物角色，在分镜的任何画面元素（气泡、思维气泡、画中画、角标、关系图示）中禁止用 emoji 字符替代，必须使用该角色的 `mini_icon_prompt` 描述渲染成头像简笔画。② prompt 文本字符串里严禁出现任何 emoji unicode 字符本身（详见规则 13a）——所有原 emoji 概念必须以"颜色 + 形状 + 手绘"的中文自然语言描述。
   - **被禁的通用 emoji 示例**：替代具体公主的 emoji、替代具体食人魔/妖怪的 emoji、替代具体王子的 emoji、替代具体国王/王后的王冠 emoji（但作为孤立的"王冠图标"指代抽象权力概念时允许，必须画成"金色王冠手绘"而非渲染 emoji 字符）、替代具体人类的 emoji、替代具体猫/狮子的动物 emoji 等——**所有故事内具体角色一律禁用 unicode emoji 字符**，必须画成头像简笔画。
   - **允许的抽象概念/物品图标**：纯抽象概念或通用物品，且不指代故事内任何具体角色——但**必须画成手绘小图标**（如"细碎鎏金星芒光点""粉红色爱心手绘""黄色小灯泡手绘"等），**严禁直接渲染 emoji 字符进画面**。允许的概念/物品：星芒光点（美丽/闪光氛围）、爱心（喜爱情绪）、小灯泡（灵光/想到主意）、水滴（汗滴/紧张）、感叹号、问号、五角星（决心）、思维云、小乌云（失落）、靶心（锁定目标）、钱包、饼干、小鱼、小校舍、钱袋等通用物品。
   - **判定口诀**：① 任何形式的 emoji unicode 字符直接出现在 prompt 文本里 → **一律禁止**（详见规则 13a）；② 概念本身（如"美丽""喜爱""饼干"）需要在画面里出现时 → 必须画成手绘小图标，并写明形状/颜色/线条；③ 故事内具体角色 → 必须用 mini_icon_prompt 头像简笔画。
   - **写法要求**：在 prompt 中调用 mini_icon 时必须显式写出来源，例如 `"气泡内画温柔公主的头像简笔画（浅金色长卷发，头戴小珍珠发箍，浅粉色衣领，圆杏眼微笑）"`，不得偷懒写"公主图标"或"公主头像简笔画"。
16. **气泡内容必须可逐词追溯到原文（硬性）**：
   - 气泡里出现的每一个英文单词，必须能在原文该角色的台词中**找到原词或同根词**（如原文 "poor man" 可写 "Poor!"，原文 "beautiful" 可写 "Beautiful!"）。
   - **气泡词原文回查算法（硬性，必跑）**：每写一个气泡 → ① 把气泡里所有英文单词列出；② 去原文该说话人的对应台词里逐词 ctrl+F 检索；③ 找不到原词或同根词的单词必须删除或替换；④ 所有气泡都做完该回查才能输出。
   - 反例：原文猫只说 "Please open the door. ... I'm a magic cat and I want to meet you."，气泡写 "Let me in!" → 原文无 "let / in"（猫从未说 let me in）→ 必须改 "Open!" 或 "Open door!" 或 "Magic cat!"。
   - 反例：原文只有 "I'm a poor man, I can't buy presents."，**不得**写 "No money!"——原文没有 money 这个词，属于归纳意译。正确做法：图标 棕色钱包手绘 + "Poor!" 或 "Can't buy!"。
   - 反例：原文 "She's very beautiful."，气泡可写 "Beautiful!"；不得写 "Pretty!"（同义词替换也算意译，必须严格用原词）。
   - 这条规则是"对白真实性"的强化版：不仅说话人不能错（不张冠李戴）、语气不能改，**用词也必须严格抽自原文**。
17. 画面英文元素必须从原文摘取或用通用英文短词，**整体追求"极简关键词"而非完整对白**：
   - **对白气泡：硬性上限 3 词**，强烈优先 1-2 词或图标+单词；只摘取最关键的情绪/动作/称呼/感叹词。
     · 例：原文 "Oh, I'd love to come in, thank you!" → "Come in!" 或 "Thanks!"。
     · 例：原文 "I really don't want to go to school today!" → "No school!" 或 红顶小校舍手绘+"No!"。
     · 优先感叹型短句（"Look!"/"Wow!"/"Run!"/"Help!"/"Yes!"/"No!"）、称呼（"Mom!"/"Bunny!"）、关键名词（"A gift!"/"Cookies!"）。
   - **原文短句原样保留例外（硬性）**：原文台词 ≤4 词时必须原样保留全部内容，含重复短语：
     · "Go away!" → 原样，禁止压成 "Go!"。
     · "Soon. Very soon." → **属于"同一发言者卖关子重复"模式**：必须画成上下相连的两个小气泡，**共用同一根气泡尾巴**指向同一说话者，体现"同一句话内的强调停顿"，不得画成左右分开、互不相连的两次独立发言；**也不得**合并成单气泡 "Soon!" 丢失重复节奏。允许的等价写法：单气泡内换行排版"Soon. / Very soon."。
     · "Stop the coach!" → 4 词原样保留。
     · "Please come in." → 3 词原样保留。
     · 判定：原文台词总词数本身就在上限内时，原样保留 > 进一步压缩。
   - **长对白图示化（硬性，二选一不叠加）**：原文 ≥5 词且含具体场景动作/能力/关系时，禁止只用 1-2 词气泡草草替代，必须从下列**两种方案中二选一**，**严禁三件套并存**（同一格内不得同时出现"文字气泡 + 关系图示气泡 + 画中画"全部表达同一句对白的同一份语义，否则触发规则 20 的"画面元素去重"）：
     a. **方案 A：关系图示气泡**（详见规则 15 第 3 优先级）——主体 mini_icon + 箭头 + 客体简笔画 + 关系符号（红色叉号手绘/深蓝色问号手绘/橙红色感叹号手绘），**不再额外加纯文字气泡**重复同一含义。
     b. **方案 B：纯文字气泡 + 配套画中画**——主气泡用极简关键词，配套画中画用简笔画演示具体内容，**画中画里不再叠加文字气泡里已有的同义词或同一对象的关系符号**。
     · 反例（叠加冗余）：原文 "I can't jump into the river. I can't swim!" → 同一格内同时出现 ① 文字气泡 "Can't swim!" ② 角色简笔小人在水中 ③ 红色叉号手绘 → 三件套都说"不会游泳"，**冗余**。修复（方案 A）：删文字气泡，仅留"角色简笔小人在水中 + 红色叉号手绘"关系图示气泡。修复（方案 B）：保留 "Can't swim!" 文字气泡，删红色叉号手绘；画中画里只有该角色在水中（不带任何否定符号）。
     · 反例：原文 "Can you change into a big lion?" → 只写 "Lion?" 不合格。修复：气泡画"被询问者 mini_icon + → + 大狮子简笔画 + 深蓝色问号手绘"（方案 A，仅一个关系图示气泡，不再加纯 "Lion?" 文字）。
     · 反例：原文 "X is in the river and he can't swim!" → 只写 "Help!" 漏两层语义。修复（方案 B）：① 主气泡 "Help!" + ② 配套画中画 X 在河里挣扎（画中画里不再加红色叉号手绘，因为"挣扎"姿态已传达"can't swim"）。
   - **思维气泡：硬性上限 2 词**，优先纯图标或图标+单词（如 圆形棕色饼干手绘+"Yummy!"）。
   - **拟声词大字**：单个全大写词（BOOM!/POP!/CRASH!/KNOCK!/WHOOSH!）。
   - **数量标记**："× 6" 等，仅 N≥5 或画面装不下时使用；≤4 必须画对应数量实物。
   - **时间标签：硬性上限 4 词**（"Day 3"/"Next morning"/"Three days later"）。
   - **物品/角色英文标签**：单词（"key"/"map"/"Bunny"）。
17a. **气泡组装规范（硬性，最高优先级，所有气泡都必须通过本规则自检）**：

   **【判定标准（孩子复述视角）】** 孩子盖住主画面只看气泡内容，能否独立还原原文该处的"WHO does WHAT to WHAT, with WHAT TONE"？还原不了即不合格。

   **【A. 气泡数量与归属】**
   - **单角色头顶气泡数 ≤1（硬性）**：同一角色同一格内，头顶/旁侧的对白气泡数 ≤1。以"独立气泡外框 + 独立尾巴"为单位计数，1 个外框内分行/分序号不算多个气泡。原文同一发言中含多语义层（多评价对象 / 评价+疑问 / 多并列陈述）时，必须合并到单个对白气泡内。禁止在同一角色头顶画 2 个及以上独立气泡（独立气泡 = 各自有独立外框且气泡尾巴各自独立指向同一说话者）。
   - **思维气泡与对白气泡分立**：思维气泡（云状外框）不计入对白气泡上限，但思维气泡同样 ≤1 个，且不得与对白气泡表达同一语义（详见规则 20）。
   - **拟声词大字与气泡分立**：拟声词大字不属气泡，不计入此上限。
   - **多语义分散写法（备选）**：除"单气泡内分行"外，可把不同语义层贴到不同载体上（评价物品的气泡贴在物品旁，评价人物的气泡贴在被评价人物 mini_icon 或画中画旁），保证每个角色/物品旁仍只有 1 个气泡。

   **【B. 元素维度分工】** 每个气泡内部的元素按 4 个语义维度划分，**每个维度只能由一个元素承担，不得两个元素承担同一维度**：
   - **D1 对象维度**：动作/态度指向"什么对象/什么概念"——���标（简笔画/mini_icon）**或**名词单词，二选一。
   - **D2 动作维度**：在做/想做"什么动作或处于什么状态"——动词/形容词单词。
   - **D3 语气维度**：陈述/疑问/感叹的句型标识——单词末尾的标点（? / ! / .）**或**语气图标（深蓝色问号手绘 / 橙红色感叹号手绘 / 红色叉号手绘），二选一。
   - **D4 情绪维度**：说话者当下情绪标识——装饰图标（粉红色爱心 / 淡黄色星星 / 红色怒气筋 / 浅蓝色水滴 / 深灰色乌云 / 黄色灯泡 / 金黄色五角星）。

   **【C. 维度重复禁令（硬性）】** 同一气泡内禁止两个元素承担同一维度：
   - D1 重复反例：小鱼简笔画 + "Fish!"（图标和单词都说"鱼"）→ 删一者，或改"小鱼简笔画 + 'Yum!'"。
   - D3 重复反例（双语气符号）：深蓝色问号手绘 + "Come?"（手绘问号 + 单词内 ?）→ 改纯 "Come?" 或加 D1 对象图标"舞会简笔画 + 'Come?'"；橙红色感叹号手绘 + "Help!" → 改纯 "Help!" 或换求救手绘+"Help!"；红色叉号手绘 + "No!" → 改纯 "No!"。
   - D4 重复反例：粉红色爱心手绘 + "Love!"（图标和单词都说"喜爱"）→ 改 粉红色爱心手绘 + "Yes!" 或仅留爱心；黄色灯泡手绘 + "Idea!" → 改 黄色灯泡手绘 + "Got it!"；红色怒气筋手绘 + "Angry!" → 改 怒气筋手绘 + "Bad!"。

   **【D. 语义骨架完整性（硬性）】** 气泡所有元素加起来必须组成最小语义骨架 `[主语] + 动作/状态 + [对象] + [语气]`：
   - 主语由说话者本人承担（可省）；动作/状态、对象、语气**必须在气泡里能找到承担元素**，三选一缺一即不合格。
   - **带具体宾语的动词（命令式祈使句、行动宣告、索取请求、寻物句、行动疑问句等）**：动作 + 对象**两者都必须可见**——要么"对象图标 + 动作单词"，要么写成"动作+对象"二元复合短语（如 "Wash clothes!" / "Cook dinner!"）。
   - **仅当宾语已在主画面被显式画出且角色明确朝向该对象**时，气泡内可省略宾语图标。
   - 反例与修复对照（举一反三，类似句式一律按此处理）：
     · "Pick up our clothes!" → "Pick up!" 裸动词 ❌ → 衣服堆简笔画 + "Pick up!" ✅ 或 "Pick up clothes!" ✅
     · "Wash the clothes!" → "Wash!" ❌ → 衣服简笔画 + "Wash!" ✅ 或 "Wash clothes!" ✅
     · "Cook the dinner!" → "Cook!" ❌ → 盘子简笔画 + "Cook!" ✅ 或 "Cook dinner!" ✅
     · "Clean the kitchen!" → "Clean!" ❌ → 扫帚简笔画 + "Clean!" ✅ 或 "Clean kitchen!" ✅
     · "Find my blue shoes." → "Find!" 或 "Shoes!" ❌ → 蓝鞋子简笔画 + "Find!" ✅
     · "Where's my ring?" → "Where?" ❌ → 戒指简笔画 + "Where?" ✅
     · "Help me with my hair." → "Help!" ❌ → 梳子简笔画 + "Help!" ✅
     · "Can I come to the ball?" → 深蓝色问号手绘 + "Come?" ❌（双问号 + 无对象，双重违规）→ 舞会场景简笔画（皇宫塔楼/水晶吊灯）+ "Come?" ✅
     · "I want to go to the ball." → "Want!" 或 "Ball!" ❌ → 舞会简笔画 + 角色指向手势 + "Go!" ✅

   **【E. 行数 = 独立语义单元数（硬性）】** 气泡分行的唯一目的是承载多个独立语义单元，不是装饰、不是布局美观、不是把"图标和文字分开放"：
   - **1 行（默认）**：气泡承载 1 个独立语义单元（同一对象 + 同一动作/状态 + 同一语气 + 同一情绪方向）。所有元素横向排在同一行。
   - **多行/多序号**：气泡承载 ≥2 个独立语义单元（不同对象 / 评价层+疑问层 / 多并列陈述）。每行 = 一个独立语义单元，每行可有自己的图标 + 单词，行间可加 ① ② ③ 或短横线分隔。
   - **独立语义单元判定**：
     · 同对象 + 同情绪方向（答应+赞许、感谢+喜爱、害怕+紧张）→ **1 单元，禁止分行**，同方向叠加只保留更具体的 1 词。
     · 不同对象（评价物品 + 评价人物）→ 多单元，分行。
     · 不同语义层（评价 + 疑问 / 陈述 + 行动宣告）→ 多单元，分行。
   - 反例与修复：
     · 头顶超限 ❌：国王头顶 3 个独立气泡 "鱼+Beautiful!" / "Daniel mini_icon+Kind!" / "?+Meet when?" → 必须合并为 1 个气泡分 3 行 ① 鱼简笔画+"Beautiful!" ② Daniel mini_icon+爱心+"Kind!" ③ 问号手绘+"Meet when?"。
     · 头顶超限 ❌（复合句）：男主头顶 2 个独立气泡 "Kind!" + "Meet when?" → 合并为 1 个气泡分 2 行。
     · 单语义拆行 ❌：气泡分两行 ① 粉红色爱心手绘 ② "Yes! Good!"（单语义答应+赞许同对象同方向，被强行拆两行）→ 修复：1 行写"粉红色爱心手绘 + 'Yes!'"，删冗余 "Good!"。
     · 单语义拆行 ❌：分两行 ① 淡黄色星星 ② "Beautiful!" → 修复：1 行"淡黄色星星手绘 + 'Beautiful!'"。

   **【F. 输出前 4 步自检（硬性，必跑）】** 每写完一个气泡立即跑以下 4 步，任一不合格必须重写：
   - ① **骨架检查（对应 D 节）**：动作、对象、语气三要素是否都能找到承担元素？带宾语的动词是否漏了对象？
   - ② **维度重复检查（对应 B/C 节）**：扫描每个气泡 D1/D2/D3/D4 四维度，是否被 ≥2 个元素重复承担？特别检查：
     · 单词带 ? → 是否还有问号手绘？（D3 语气重复）
     · 单词带 ! → 是否还有同语义感叹手绘？（D3 语气重复）
     · 名词单词 → 是否还有同物体图标？（D1 对象重复）
     · 情绪形容词单词 → 是否还有同情绪装饰图标？（D4 情绪重复）
   - ③ **行数检查（对应 E 节）**：行数是否等于独立语义单元数？同对象同情绪却拆行 → 必须合 1 行；多对象却塞 1 行 → 必须分行。
   - ④ **气泡数检查（对应 A 节）**：同一角色头顶/旁侧独立对白气泡 ≤1 个，思维气泡 ≤1 个。

18. 禁止画面出现中文、无关长段英文、虚构乱码。**禁止把原文整句对白照搬进气泡**；如果一句原文承载了多重信息（情绪 + 动作 + 称呼），优先只挑一项最关键的做成短词气泡，剩下的靠角色表情、肢体动作、构图分格去表达。
19. **对白真实性（硬性）**：气泡文字严格忠于原文事实：
   - 对白气泡只能给原文真实开口说话的角色。原文只写"微笑/点头/看着"等非语言动作时禁加对白气泡（用表情/肢体/装饰符号代替）。反例：原文 "X smiles at Y." 给 X 加 "Hi!" → 必须删（原文 X 没说话）。
   - 气泡文字必须从原文该角色实际台词浓缩，不得换成原文里没出现过的语义/情绪。反例：原文角色说 "Yes, I do. She's very beautiful." 写 "Like her!" → 原文无 like → 必须改 "Yes!"/"Beautiful!"/"I do!"。
   - **不得张冠李戴**：每条气泡严格归属原文中说该话的角色，不得挪移或合并多人台词。反例：原文角色 A 说的台词被错误归到角色 B 头上 → 必须归还原说话角色。
   - **无 says 标注的对话归属推断（硬性）**：原文连续多段引语中仅部分有 `says X` 标识时，未标识引语的归属必须按"交替原则 + 语义自洽校验"两步推断：
     · **交替原则**：英文绘本中无 says 标注的引语默认接续对方角色（A 说 → B 说 → A 说 → B 说…），不得简单挂到上一个 says X。
     · **语义自洽校验**：归属人不能是自己评价的对象（"你的女儿很美"不能由女儿的父亲说出）；归属人对当前听众的称呼/角度必须合理（"Your Majesty" 暗示说话者是平民对国王说，反之亦然）。
     · 反例：原文 `"...beautiful castle," says the King. / "Thank you," says Daniel. / "And you've got a very beautiful daughter."` → 第三句无 says，按交替原则下一位回到国王，但**语义校验**：公主是国王的女儿 → 国王不可能对自己说"你有美丽的女儿" → 必须归 Daniel。气泡尾巴必须指向 Daniel，不得指向国王。
     · 自检：写每条对白气泡前，先回查原文该句是否有 `says X` 标注；无标注时跑"交替 + 语义"两步算法，输出归属人写在 prompt 注释里（如"该句归 Daniel，原文无 says 标注，按交替+语义推断"）。
   - **不得改变语气**：陈述保持陈述，疑问保持疑问。反例：原文陈述 "I want to marry her." 写成疑问 "Marry her?" → 必须改 "Marry her!"/"Want to marry!"。
   - 思维气泡仅在原文明确写心理活动（"X thinks/wonders/wants/is afraid"）时使用，且不得重复说出口的内容。
   - 拟声词、数量标记、时间标签、物品标签不属对白，但必须与原文事实一致。
20. **画面元素去重（硬性）**：同一角色同一格内，多个元素不得表达同一份信息：
   - 已用对白气泡说"想到主意"，禁止再用思维气泡/小灯泡重复。
   - 已用表情和肢体表达"得意"，禁止再加思维气泡 "Smart!"。
   - **图标+单词维度重复**：图标和单词不得指代同一对象。反例：小鱼手绘图标 + "Fish!"（两者都说"鱼"）→ 重复。修复：删图标或单词；或改"小鱼手绘图标 + 'Catch!'/'Yum!'"。合法：钱包手绘图标 + "Poor!"（钱包对象 + 态度）、饼干手绘图标 + "Want!"（饼干对象 + 欲望）、公主 mini_icon + "Meet?"（对象 + 疑问）。判定：图标承担"是什么"，单词承担"怎么了/想怎样"，二者不能都回答同一问题。
   - **维度重复（含语气/情绪/对象/动作四维度）**：完整规则详见 17a 气泡组装规范的 B/C 节维度分工与重复禁令。本条规则 20 仅作快速速查：图标和单词不得指代同一对象；语气符号不得双叠（"?"+问号手绘 / "!"+感叹号手绘均禁）；情绪图标不得与同义情绪单词共存（爱心+Love! / 灯泡+Idea! 均禁）。
   - **态度/动作词必须显式标出对象**：角色说带宾语的态度词（喜欢/不喜欢/害怕/想要）或带宾语的动作词（洗/煮/扫/找/拣/给）时，气泡必须含态度/动作词 + 对象指代（mini_icon / 物品图标 / 二元复合短语 "Verb+Noun"），不得只抽动词或态度词。详细判定与修复对照详见规则 17a 气泡组装规范的 D 节语义骨架完整性。
   - 元素叠加合法情形：每个元素承载不同维度（对白+拟声词+思维+画中画）。
   - 自检：删掉某气泡/图标/标签是否丢失原文信息？不丢则必须删。

## 5.3.5 分镜校验节点

### 节点用途

在 5.3 输出分镜后，用一个**独立提示词的 LLM** 对分镜做对齐校验，并**直接输出修正后的 storyboard_assets**（原格式、原字段、原 id 编号规则）。本节点完成"校验 + 修复"一体化，不输出问题清单，不需要后续 5.4 介入即可得到可用于生图的最终分镜。

### 与 5.3 的边界

- 5.3 **生成** storyboard_assets，关注"如何把原文拆成分镜"——生成阶段的注意力容易分散，常出现漏句、张冠李戴、构图 ID 泄漏等问题。
- 5.3.5 **专项扫描器**（不是"再跑一遍 5.3"）：只做三类高频违规专项扫描——① 漏句扫描；② 气泡组装合规扫描（17a 的 4 步自检）；③ 角色/场景一致性扫描。**凡 5.3 已经做对的部分，原样保留，禁止改写**。
- 两个节点使用**完全不同的系统提示词**，5.3.5 不得包含 5.3 的生成思路，必须站在"孩子复述视角 + 质检视角"独立判断。
- 5.3.5 **可以**修改 storyboard_assets 的 prompt 文本、character_refs、scene_refs；**可以**新增分镜、删除分镜、合并分镜、拆分分镜、调整分镜顺序；**不得**修改 character_assets 和 scene_assets（这两个由 5.1 / 5.2 负责）。
- **最小变更原则（强化）**：未命中三类专项扫描的分镜，prompt 字符串保持原样（仅 id 可能因重编号而变）。即使你认为某个 prompt"可以写得更好"——只要它不违规，就**禁止**改写。

④ 漏覆盖必须新增分镜或在现有分镜的 prompt 中增加图层（画中画 / 思维气泡 / 关系图示气泡 / 角标）补回。

特别关注的高频漏覆盖句型（命中即修）：
- **空间关系句**（"X is near Y"）→ 同框出现两者，**near ≠ adjacent**：用中间间隔物（田野/缓坡/小路/树篱）承载视距感，禁用"挨得近/紧邻/相邻"等贴边词。
- **群体状态句**（"everyone in the village frightens"）→ 显式画 3-5 个村民群像 + 具体反应，不得用主角单句态度台词代替。
- **心理动机句**（"X wants to meet Y"）→ 思维气泡或眼神体现动机。
- **状态描写句**（"X smiles at Y"）→ 必须独立成格或独立分格。
- **评价性形容词句**（beautiful/handsome/pretty/kind/lovely/wonderful 修饰人物）→ 被评价对象身上必须叠加视觉装饰（详见 5.3 规则 13b 的 AURA 短语库直接调用），不得仅靠赞叹方表情承载。
  · 美丽（人物）：调用 AURA-BEAUTY-1（角色周身萦绕细碎鎏金星芒光点，伴随淡金色朦胧星屑飘散）+ 飘发 + 脸颊泛粉 + 花瓣（≥2 项）
  · 帅气：调用 AURA-CHARM（身后鎏金流光粒子拖尾，细碎金芒漫散）+ 挺胸 + 自信姿态（≥2 项）
  · 善良：ICON-LOVE 粉红色卡通爱心手绘 + 柔光圈 + 温和表情（≥1 项）
- **代词群体回溯句**（They/We/The family/Everyone/All of them）→ 向前回溯最多 5 句找到完整指代来源，**全部成员**写入该分镜 character_refs 与 prompt 主体描述。任何成员遗漏即视为漏覆盖必须补回。
- **在场名单稳定**（无场景跳变/无时间跳跃/无离场动作时）→ 前一镜在场角色必须保留到下一镜的 character_refs 与画面身位，可调整身位但不得移出 character_refs。
- **疑问/求助/请求句**（"How can I...?"/"Please help!"）→ 提问者加歪头/挠头 + 与原文一致的疑问气泡，不得省略。
- **索取/请求物品句**（"give me X"/"I want X"/"Please bring X"）→ 必须独立分镜或分格承载索取，画面要素：① 索取者伸手/摊掌；② 被索取物品的图标/画中画清晰可见；③ 气泡抽原文动词。禁用"已获得物品后的状态"代替"索取过程"。
- **复合句（评价+疑问）**（"X is very kind. When can I meet him?"）→ 两个语义层都必须独立可视化，但**承载方式必须遵守"单角色头顶气泡 ≤1"硬性上限**——默认在说话者头顶 1 个气泡内分行（① 评价层；② 疑问层），禁止开 2 个独立气泡。
- **原文短句原样保留**（≤4 词台词）→ 必须原样保留含重复短语；卖关子重复（"Soon. Very soon."）必须画成上下相连两个小气泡共用同一根气泡尾巴指向同一说话者，或单气泡内换行。"Go away!" 不得压成 "Go!"。
- **长对白具体语义图示化**（≥5 词且含具体场景动作/能力/关系）→ 必须用关系图示气泡或配套画中画（二选一，不叠加三件套）。
- **叙事高潮加密**（计谋/身份揭晓/物理冲突/情感高潮/变形）→ 高潮段允许 3-6 格，被压缩在 1-2 格内必须扩展。

═══ 第 2 类：气泡组装合规扫描（5.3 规则 17a 的 4 步自检） ═══
对每条分镜的每个气泡，依次跑 4 步：

H-1. **骨架完整**（17a-D）：动作、对象、语气三要素是否齐全？带宾语动词漏对象？
   · 高频违规：命令句"Wash the clothes!"/"Cook the dinner!"/"Pick up our clothes!"/"Clean the kitchen!" → 抽成 "Wash!"/"Cook!"/"Pick up!"/"Clean!" 裸动词（漏对象）→ 必须改"宾语图标 + 动词"或"动词+宾语"二元复合短语（如"衣服简笔画 + 'Wash!'"或 "Wash clothes!"）。
   · 高频违规：寻物句"Find my blue shoes!"/"Where's my ring?" → 抽成 "Find!"/"Where?" → 必须加物品图标。
   · 高频违规：行动疑问"Can I come to the ball?" → 抽成 "Come?" → 必须加目的地图标（皇宫塔楼简笔画 + "Come?"）。

H-2. **维度不重复**（17a-B/C）：D1 对象 / D2 动作 / D3 语气 / D4 情绪 四维度每维只能由 1 个元素承担。
   · D3 双语气违规：问号手绘 + "Come?" / 感叹号手绘 + "Help!" / 叉号手绘 + "No!" → 删一者。
   · D4 情绪重复违规：粉红色爱心手绘 + "Love!" / 黄色小灯泡手绘 + "Idea!" / 红色怒气筋手绘 + "Angry!" → 删一者或换非同义词。
   · D1 对象重复违规：小鱼简笔画 + "Fish!" / 钱包图标 + "Wallet!" → 删一者或换"图标 + 动作/态度词"。

H-3. **行数 = 独立语义单元数**（17a-E）：同对象同情绪却拆行 → 合并 1 行；多对象/多语义层却塞 1 行 → 分行排版（① ② ③ 序号）。
   · 单语义拆行违规：① 粉红色爱心 ② "Yes! Good!" → 合 1 行"粉红色爱心 + 'Yes!'"，删冗余 Good。
   · 单语义拆行违规：① 鎏金星芒 ② "Beautiful!" → 合 1 行"细碎鎏金星芒光点 + 'Beautiful!'"。
   · 多语义合 1 行违规：1 行塞"鱼+Beautiful! mini_icon+Kind! 问号+Meet when?" → 拆 3 行。

H-4. **气泡数 ≤1**（17a-A）：同一角色同一格内独立对白气泡（独立外框 + 独立尾巴）数 ≤1；思维气泡同样 ≤1 个。
   · 反例：国王头顶 3 个独立气泡 "鱼+Beautiful!" / "Daniel mini_icon+Kind!" / "?+Meet when?" → 必须合并为 1 气泡分 3 行。
   · 反例（复合句）：男主头顶 2 个独立气泡 "Kind!" + "Meet when?" → 合并为 1 气泡分 2 行。

补充几条对白真实性检查（属于第 2 类的延伸）：
- **非语言动作禁加对白**：原文 "X smiles/nods/looks at Y" 给 X 加 "Hi!" → 删除该气泡。
- **气泡词原文回查**：气泡每个英文单词必须能在原文该角色台词找到原词或同根词。反例：原文 "She's very beautiful." 写 "Pretty!" → 同义词替换也算意译 → 改 "Beautiful!"。
- **不得张冠李戴**：气泡严格归属原文真正说话角色。
- **无 says 标注的归属推断**：未标注引语按"交替原则 + 语义自洽"两步推断（详见 5.3 规则 19）。归属人不能是自己评价的对象（"你的女儿很美"不能由女儿的父亲说出，必须归对方）。
- **不得改变语气**：陈述↔疑问不互转。

═══ 第 3 类：角色/场景一致性扫描 ═══

I-1. **原文人物清单回查**：article_content 做"全员名词盘点"——列出原文所有显式命名/称谓/群体点名的人物（含代词回溯），去重得"人物清单"，与 character_assets 逐一比对。若 character_assets 缺失任何原文人物（典型：原文有 "the Queen" 但角色卡缺）→ 属上游 5.1 严重遗漏，本节点不可修改 character_assets，必须在分镜 prompt 中**用原文称谓+详细外观描述**显式补齐（如"王后：中年女性，40岁左右，浅金色盘发戴银色发冠，穿浅紫色宫廷长裙，气质温婉"），character_refs 中**不得捏造新 id**。

I-2. **角色形态时序**：变体变化前必须用主形态 id（如 C1），变化后必须用变体 id（如 C1b）；不混用或颠倒。修复：替换 character_refs，调整 prompt 中外观描述（"普通家猫" vs "穿靴戴帽的家猫"）。

I-3. **变体形态描述词（不写 id）**：当本格使用某角色变体时，character_refs 数组中填变体 id（如 ["C2b"]），但 prompt 文本里只写形态描述词（如"穿斗篷的角色"），不得写 id。

I-4. **非人形角色穿戴方位**：变体角色为四足动物穿人类服饰时，prompt 必须显式写明每件服饰的具体身体部位（如"两条后腿穿 [颜色] 高筒靴，前爪保持原本毛色裸露不穿靴"；帽子写"戴在两耳之间偏后位置，两耳完全露出"）。每次出现该变体的分镜，prompt 必须重述穿戴方位关键短语。

I-5. **多角色同框比例**：≥2 角色（含动物或体型差异显著角色）同框时必须显式写出相对高度参照（如"小动物站立时头顶仅及成人膝盖；成人蹲下时小动物头顶约到成人腰部"）。每次出现动物或特殊体型角色都要重申其标志性高度词组。

I-6. **refs 合法性 / ID 泄漏 / emoji 残留**（机械扫描）：
   · refs：character_refs / scene_refs 只能引用 character_assets / scene_assets 中已有的 id，**严禁新增或编造 id**。每条分镜必须输出这两个字段，可为空数组。
   · ID 泄漏：对每条 prompt 跑两遍正则 `\b[CSBT]\d+[a-z]?\b` 与 `\bSent\d+\b`，命中即改写为自然语言（如"T1序列条" → "序列条"）。
   · emoji 字符残留：扫描 prompt 是否含 emoji unicode 字符（U+1F300-U+1FAFF、U+2600-U+27BF），命中即替换为 13b 短语库的对应手绘描述。
   · 装饰元素分级合规：A 级氛围装饰是否调用了 AURA 短语库？B/C/D 级图标是否被错误叠加 A 级文学化修饰（如怒气筋/水滴/钱包等被加了"鎏金""星尘""光晕萦绕"等 A 级修饰词）？

【硬性修复规则】
1. 完整性：输出必须是**完整的 storyboard_assets 数组**，包含所有分镜，不是差分。
2. 原格式：每条分镜必须且只能包含 4 个字段 —— id、character_refs、scene_refs、prompt；字段顺序与 5.3 一致。
3. 重新编号：最终数组按 B1、B2、B3…… 顺序从 1 开始重新编号，不得保留跳号。
4. refs 合法性：character_refs 和 scene_refs 只能引用已有 id，**严禁新增或编造**。
5. **最小变更（强化）**：未命中三类专项扫描的分镜，prompt 字符串原样保留（仅 id 可能因重编号而变）；只修违规元素，**禁止整段重写**，**禁止**为"风格统一"或"更好"做无关改写。
6. 不得删除原文信息：合并/删除分镜时必须确认被合并/删除分镜的原文信息已被其他分镜吸收。
7. 不得越权修改：不得修改 character_assets 和 scene_assets。
8. prompt 写作规范继承 5.3：短句化、逗号分隔、重要信息前置、气泡用英文引号写出确切文本、构图用自然语言描述。
9. 画面文字规则继承 5.3：对白气泡 ≤3 词、思维气泡 ≤2 词、拟声词单个全大写词、时间标签 ≤4 词、物品/角色标签单词。
10. 输出纯 JSON，顶层只保留 storyboard_assets 字段，不含解释、评论、修改说明、问题报告。

## 5.4 分镜调整节点

### 节点用途

在 5.3 输出分镜后，用户可以查看 storyboard_assets。若觉得某段剧情跳跃过大、某几格过碎、某些画面不符合逻辑或信息不足，可以输入调整意见，本节点重新输出完整的 storyboard_assets。
角色设定和场景设定原则上不变。本节点主要用于微调分镜拆分方式、画面表达重点与 prompt 清晰度。

### 系统提示词

```text
你是儿童英文绘本分镜调整助手。根据 adjustment_instruction，在不改变原文核心剧情前提下重新整理完整 storyboard_assets，保持适配 Nano Banana 2。
只输出合法 JSON，不含解释、注释、Markdown 代码块。顶层字段只保留：storyboard_assets。
【输入】article_content、storyboard_assets、character_assets、scene_assets、adjustment_instruction。
【调整目标】
1. 优先调整用户明确提到的分镜范围。
2. 未提及分镜尽量保持原意，除非为叙事衔接、空间逻辑或角色一致性必须微调。
3. 调整后所有分镜重新从 B1 开始顺序编号。
【硬性规则】
本节点的硬性规则与 5.3 / 5.3.5 保持一致，**完整规则详见 5.3 节**。本节点的特殊性仅在"如何根据 adjustment_instruction 微调"，其他规则全部继承。
核心规则速查（与 5.3 一致）：
1. 每句原文必须能在分镜中找到对应画面；合并分镜时不得丢失原文信息。
2. 节拍而非动作：单镜内可同时含主动作 + 对白 + 心理 + 关键道具 + 数量/时间标记。高密度叙事，鼓励多格、气泡、画中画、角标。
3. 场景跳变才拆镜；同场景两角色一来一回连续对话必须用对话双格合并 1 格（详见 5.3 规则 4 分镜密度原则）；单格内对话回合 ≤2，气泡总数 ≤3，超出按角色态度转折点拆格；横跨"拒绝→接受/怀疑→相信/抗拒→反转"语义转折禁止合并（详见 5.3 规则 5 必拆禁区）。
4. **叙事高潮加密（硬性）**：识别 5 类高潮节点（计谋诡计执行/身份揭晓/物理冲突/情感高潮/变形过程）并加密，每节点 ≥2 格、复杂可达 5 格。详见 5.3 规则 6 叙事高潮加密；计谋类桥段同时受 5.3 规则 5.3 计谋反转禁区（≥4 格）约束，取较严下限。
5. 构图主动选择：每条分镜从构图配方表选 1 种主构图，prompt 用自然语言写出。一级触发都不命中时优先 C2 兜底。**对话双格双方都可见 + ≥3 角色禁用对话双格**（详见 5.3）。
6. **角色/场景一致性**：完整外观/空间结构由参考图承担，prompt 不全文复制。**非人形角色穿戴方位**：四足动物穿人类服饰必须显式写明每件服饰的具体身体部位（如四足动物穿靴必须写"两条后腿穿靴，前爪保持原本毛色裸露"；戴帽必须写"戴在两耳之间偏后位置，两耳完全露出"）。**多角色同框比例**：≥2 角色同框时必须显式写出相对高度参照（如"小动物站立时头顶仅及成人的膝盖"），每次出现动物或特殊体型角色都要重申其标志性高度词组。**在场名单稳定（硬性）**：同场景同时间段（无场景跳变 / 无时间跳跃 / 无离场动作）下，前一镜在场角色必须保留到下一镜的 character_refs 与画面身位，不得无故消失（详见 5.3 规则 9a）。**代词群体回溯（硬性）**：原文出现 They/We/Everyone/The family 等代词时，回溯前 5 句找到完整指代来源，全部成员必须出现在 character_refs 与画面（详见 5.3 规则 2 代词群体回溯句）。
7. character_refs 和 scene_refs 只能引用已有 id，严禁新增。
8. **prompt 文本内禁止任何内部 ID（硬性）**：资产 ID（含变体）C1/C2/C2b/C3a/S1/S1b/S3 等、构图配方 ID T1/T2/D1/C2/M1 等、内部句子编号 Sent01/Sent02 等都禁；构图必须用自然语言（"序列条横排三格构图"而非"T1序列条"）；变体形态用纯描述词（"穿靴戴帽的橘猫"），不通过 ID 表达。输出前必须正则扫描 `[CSBT]\d+[a-z]?` 与 `Sent\d+`。
9. **画面文字规则**（详见 5.3 规则 14-21）：
   - 对白气泡 ≤3 词；思维气泡 ≤2 词；拟声词单个全大写词；时间标签 ≤4 词；物品/角色标签为单词。整体追求"极简关键词"而非完整对白。
   - **单角色头顶气泡上限（硬性，最高优先级）**：同一角色同一格内独立对白气泡（独立外框 + 独立尾巴）数 ≤1。多语义层（多评价对象 / 评价+疑问 / 多并列）必须合并到 1 个气泡内分行 / 用 ① ② ③ 序号承载。**禁止**同一头顶画 2 个及以上独立气泡（详见 5.3 规则 17a）。
   - **图像优先（硬性）**：能用图像、表情、肢体表现的信息禁用文字气泡。
   - **气泡形式优先级**：①纯图标/图示气泡 → ②图标+原文单词 → ③关系图示气泡（"主体 mini_icon+箭头+客体简笔画+关系符号"）→ ④单一原文关键词。禁用整句意译归纳。
   - **气泡内容必须可逐词追溯到原文**：每个英文单词必须能在原文该角色台词找到原词或同根词，不得意译/同义词替换。
   - **emoji 字符渲染禁令（硬性）**：① 故事内已建角色卡的人物/动物（公主、食人魔、王子、国王、人类、猫、狮子等）画面任何位置禁用 emoji 字符替代，必须用该角色的 mini_icon_prompt 简笔画头像；② prompt 文本字符串里禁止出现任何 emoji unicode 字符（包括抽象概念如星星、爱心、叉号、问号等），这些必须改写为"颜色 + 形状 + 手绘"的中文描述（如"细碎鎏金星芒光点""粉红色爱心手绘""红色叉号手绘""深蓝色问号手绘"等）。详见 5.3 规则 13a。
   - **原文短句原样保留（硬性）**：原文台词 ≤4 词时必须原样保留，含重复短语（同一说话人卖关子重复 "Soon. Very soon." 必须画成上下相连两小气泡共用同一根气泡尾巴指向同一说话者，或单气泡换行；不得左右分开成两次独立发言；"Go away!" 不得压成 "Go!"）。
   - **长对白图示化（硬性，二选一不叠加）**：原文 ≥5 词且含具体场景动作/能力/关系时，必须二选一——方案 A 关系图示气泡（mini_icon+箭头+客体+关系符号）或方案 B 极简文字气泡+画中画。**严禁**同一格内"文字+简笔画+关系符号"三件套都表达同一份语义（如 "Can't swim!" + 水中小人 + 红色叉号手绘 三层冗余）。
   - **索取/请求物品句独立成镜（硬性）**：必须有独立分格画索取过程（伸手/摊掌+物品图标+原文动词气泡）。
   - **复合句双语义层独立可视化（硬性）**：原文同时含评价+疑问时两层都必须独立可视化，但承载方式必须遵守"单角色头顶气泡 ≤1"——默认在说话者头顶 1 个气泡内分行（① 评价层；② 疑问层），禁止开 2 个独立气泡叠在同一头顶。
10. **对白真实性（硬性）**：①对白气泡只给原文真实开口说话的角色（"smiles/nods/looks at" 等非语言动作禁加对白气泡）；②气泡文字必须从原文该角色实际台词浓缩，不得换成原文里没出现过的语义；③不得张冠李戴；④**无 says 标注的对话归属推断（硬性）**：原文连续多段引语中部分有 `says X` 标注时，未标注引语按"交替原则 + 语义自洽校验"两步推断归属——不得简单挂到上一个 says X；归属人不能是自己评价的对象（如"你的女儿很美"不得由女儿的父亲说出，必须归对方）。详见 5.3 规则 19；⑤不得改变语气（陈述↔疑问）；⑥思维气泡仅在原文明确写心理活动时使用。
11. **气泡组装规范（硬性，最高优先级）**：所有气泡必须通过 4 步自检——① 骨架完整（动作+对象+语气三要素齐全；带宾语动词必须显式带对象，不得抽成裸动词如 "Wash!"/"Cook!"/"Find!"）；② 维度不重复（D1 对象/D2 动作/D3 语气/D4 情绪每维只能由 1 个元素承担，禁止"问号手绘+?"双语气、"爱心+Love"双情绪、"鱼图标+Fish"双对象）；③ 行数 = 独立语义单元数（同对象同情绪禁拆行，多对象/多语义层禁塞 1 行）；④ 同角色头顶气泡数 ≤1（详见 5.3 规则 17a 气泡组装规范）。
12. 所有要在画面上的文字必须在 prompt 中用引号写出确切英文。
